Problem-Solving Reflection

1. Blocked on a Simple Problem

What was the problem?

There was a time I was building a simple web app, and I spent hours trying to figure out why my button wasn’t working. The app was supposed to add items to a list when the button was clicked, but nothing happened.

What problem-solving techniques did you use?

I went back and reviewed my code line by line, but still couldn’t see the issue. I tried Googling solutions, checked online forums, but no luck. I even used the "rubber ducky" method, where you explain your problem out loud. Finally, I checked the console and found a small typo in my button’s JavaScript function—I'd written "addItmes" instead of "addItems".

How did you feel throughout the process?

At first, I was confused. As time went on, frustration started to build, and I felt like the problem was much more complicated than it actually was. When I finally spotted the typo, I laughed at how simple the solution was and felt a sense of relief.

What did you learn?

I learned that small details can cause big problems in code and that checking the console for error messages early on could save a lot of time.

2. Elegantly Solved a Problem

What was the problem?

I was developing a feature for a client’s website where they wanted a search bar that filtered results as the user typed. With a large database, I thought this might be tricky.

What problem-solving techniques did you use?

I used a technique called "debouncing" to prevent the search function from sending too many requests at once. I also paginated the results to avoid overloading the webpage with too much data at once.

How did you feel throughout the process?

I felt confident because I had encountered similar issues before, and I knew that performance optimization was the key. It was incredibly satisfying when the search worked smoothly and responsively.

What did you learn?

I learned how useful debouncing and pagination are for improving the performance and user experience of a website.

3. Reflection on Problem-Solving Techniques

  • Pseudocode: Fairly confident. It helps break down complex problems, but I don’t always use it for smaller tasks.
  • Trying Something: Very confident. Experimenting is often key in finding a solution when stuck.
  • Rubber Ducky Method: It works surprisingly well. Talking through problems helps clarify my thinking.
  • Reading Error Messages: Very confident. Error messages usually point me in the right direction.
  • Console.logging: I use this all the time to track what’s happening in my code and troubleshoot effectively.
  • Googling: Extremely confident. Google is a great resource, and someone else has usually encountered the same issue.
  • Asking Peers for Help: Becoming more comfortable with this, realizing that collaboration is a learning opportunity.
  • Asking Coaches for Help: I hesitate sometimes, thinking I should figure things out alone, but I’m working on this.
  • Improving Your Process with Reflection: Reflection helps refine my approach for future problems. I’m learning to appreciate its value.

4. Reluctance to Ask for Help

A time you were reluctant to ask for help:

When I was stuck on that button issue, I didn’t want to ask for help because it seemed so small. I felt I should be able to solve it on my own, especially since it was basic.

What made you reluctant?

I didn’t want to admit that I was struggling with something simple. I was worried it might make me seem less capable, and I also didn’t want to bother anyone with an easy problem.

What might you try differently next time?

Next time, I’ll remind myself that everyone gets stuck sometimes, and asking for help—even on small issues—is completely fine. It’s better to solve the problem quickly than waste time. I’ll also see asking for help as a learning opportunity.
